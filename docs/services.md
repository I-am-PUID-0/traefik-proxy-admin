# Service Configuration

Services are the primary objects managed by Traefik Proxy Admin. A service combines a public hostname rule, a target upstream, optional authentication, optional middlewares, and generated Traefik router/service config.

## Service Basics

A standard service defines:

- Service name
- Domain and hostname mode
- Target IP or host
- Target port
- HTTP or HTTPS upstream scheme
- Optional entrypoint override
- Optional middleware list
- Optional request headers
- Optional auto-disable duration

Generated service config is exposed through:

```text
GET /api/traefik/config
```

Disabled services are excluded from generated Traefik config.

## Domains

Domains define reusable base domains, certificate resolver settings, and wildcard certificate behavior. Services attach to a domain and generate host rules from the selected hostname mode.

Use the Domains page to manage domain records before creating services that should share certificate behavior or default routing settings.

## Middlewares

Service middlewares are stored as Traefik middleware references such as:

```text
chain-oauth@file
secure-headers@file
```

If `TRAEFIK_API_URL` is configured, the service form can discover live Traefik middlewares and offer them in a selector. Manual middleware entry remains available even when live discovery is disabled.

Global middlewares apply before service-specific middlewares. Authentication middlewares generated by TPA are inserted when service security rules require them.

## Advanced Service Rules

The normal form covers the common router and load-balancer path. Advanced fields are available when a service needs more precise Traefik config.

### Pass Host Header

`passHostHeader` controls whether Traefik sends the original public Host header to the upstream. Disable it when the target service must receive the target host instead of the public route hostname.

### Managed Middlewares JSON

Managed middleware JSON creates middleware definitions under `http.middlewares` in the generated Traefik config. Use this for app-managed middlewares that belong to one service.

Example redirect middleware:

```json
{
  "redirect-to-admin": {
    "redirectRegex": {
      "regex": "^https?://pihole.example.com/$",
      "replacement": "https://pihole.example.com/admin",
      "permanent": true
    }
  }
}
```

### Bypass Rules

A service Security page can create **Bypass Rule** configurations for common auth exceptions such as mobile apps, webhooks, health checks, or automation callbacks. TPA generates a higher-priority router for the same backend service and leaves the normal SSO/shared-link/basic-auth router in place.

Bypass modes:

- **Simple bypass** forwards matching traffic without touching TPA auth endpoints. It is the most isolated option and does not create Sessions rows.
- **Observed bypass** adds an allow-only TPA forwardAuth observer before the service. It still bypasses authentication, but records lightweight access metadata in Sessions while the bypass rule remains enabled.

Bypass rules accept Traefik match expressions such as `HeaderRegexp(...)`, `Header(...) || Query(...)`, `Method(...) && Path(...)`, `PathPrefix(...)`, or `ClientIP(...)`. If the rule does not include `Host(...)`, TPA adds the service host rule automatically. Set a high priority, usually `100`, so the bypass route wins over the normal protected route. Optional bypass middlewares can keep rate limits or headers on the bypass lane. Use `@file` only for middlewares that still live in Traefik's file provider; TPA-managed middlewares are referenced without a provider suffix.

### Additional Routers JSON

Additional routers point at the same generated backend service. Use them for alternate match rules, explicit router priority, entrypoint overrides, middleware overrides, or cert resolver overrides. Prefer Security page Bypass Rules for auth bypasses when possible.

Example bypass router:

```json
[
  {
    "name": "api-bypass",
    "rule": "Host(`app.example.com`) && Header(`X-Api-Key`,`REDACTED`)",
    "entrypoint": "websecure",
    "middlewares": [],
    "certResolver": "cloudflare",
    "priority": 100
  }
]
```

Set `middlewares: []` only when the additional router should intentionally bypass the normal middleware chain.

## Import And Export

Services can be exported individually from a service row or all at once from the Services header.

The export format is portable JSON:

```text
traefik-proxy-admin.services v1
```

Exports include routing fields, domain metadata by domain name, middleware references, request headers, managed middleware JSON, and advanced routers. Exports omit database IDs, timestamps, shared links, SSO rules, Basic Auth users, and other security state.

Imports can create missing domains. When a service name or subdomain conflicts with an existing service, choose whether to rename the imported service or skip it.

## Target Tests

The service form can test TCP reachability to a target host and port. In production, target probes are disabled unless `TARGET_TEST_ALLOW_CIDRS` is set. See [Security Hardening](security-hardening.md#target-probes) before enabling probes.
